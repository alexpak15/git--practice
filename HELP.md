    GitHub Pages — это служба размещения статических сайтов, которая принимает ФАЙЛЫ HTML, CSS и JavaScript прямо 
        из репозитория на GitHub, при необходимости выполняет файлы через процесс сборки и публикует веб-сайт.

    Типы сайтов GitHub Pages
        Существует два типа сайтов GitHub Pages. Сайты, связанные с учетной записью пользователя или организации,
        и сайты для определенного проекта.

Свойство	            Сайты пользователей и организаций	            Сайты проектов

                        <owner>.github.io,                              содержащего код проекта 
                        где <owner> имя личной или                      (обычно docs/ или gh-pages branch)
                        учетной записи организации
                                   

Ограничения	            Не более одного сайта страниц	                 Не более одного сайта страниц
                        на одну учетную запись                           на каждый репозиторий

    Размещение в собственном пользовательском домене
Сайт можно разместить на GitHub github.io домена или личного домена. Пользователи с правами администратора репозитория 
могут настроить собственный домен для сайта GitHub Pages.
    
    Использование пользовательского домена в нескольких репозиториях
Если вы настроите пользовательский домен для сайта пользователя или организации, по умолчанию он будет 
использоваться для всех сайтов проектов, принадлежащих этой учётной записи. 
Например, если настраиваемый домен для вашего сайта пользователя — www.octocat.com, и у вас есть сайт проекта 
без настроенного настраиваемого домена, опубликованный из репозитория с именем octo-project, 
сайт GitHub Pages для этого репозитория будет доступен по адресу www.octocat.com/octo-project.
Вы можете переопределить пользовательский домен по умолчанию, добавив пользовательский домен в отдельный репозиторий.
Примечание: URL-адреса сайтов проектов, опубликованных в частном порядке, не зависят от настраиваемого домена 
вашего сайта пользователя или организации.
    
    Изменение видимости вашего сайта GitHub Pages
Вы можете управлять контролем доступа к сайту своего проекта, опубликовав сайт публично или конфиденциально.

    О контроле доступа к сайтам GitHub Pages
С помощью управления доступом к GitHub Pages вы можете ограничить доступ к сайту своего проекта, 
опубликовав его в частном порядке. Доступ к сайту, опубликованному в частном порядке, 
могут получить только пользователи, имеющие право на чтение репозитория, из которого он был опубликован. 
Вы можете использовать сайты, опубликованные в частном порядке, 
для предоставления доступа к внутренней документации или базе знаний сотрудникам вашей организации.
Примечание: Чтобы опубликовать сайт GitHub Pages в частном порядке, ваша организация 
должна использовать GitHub Enterprise Cloud. 
https://github.com/account/enterprises/new?ref_product=ghec&ref_type=trial&ref_style=button&ref_plan=enterprise
Если ваша компания использует права Enterprise Managed Users, сайты GitHub Pages 
могут быть опубликованы только как частные, а все сайты GitHub Pages доступны только другим участникам компании. 
Если ваша организация использует GitHub Enterprise Cloud без Enterprise Managed Users, 
вы можете публиковать сайты своих проектов конфиденциально или публично для любого пользователя в Интернете.
Управление доступом доступно для сайтов проектов, опубликованных из частного или внутреннего репозитория, принадлежащего организации.
Вы не можете управлять доступом для сайта организации.
Чтобы удалить пользовательский домен по умолчанию, необходимо удалить пользовательский домен с сайта пользователя или организации.

    О настройке пользовательского домена
Совет: Мы рекомендуем проверить ваш пользовательский домен перед добавлением его в репозиторий, чтобы повысить безопасность 
и предотвратить атаки с целью его перехвата.
Обязательно добавьте свой пользовательский домен на сайт GitHub Pages, прежде чем настраивать его у DNS-провайдера. 
Настройка пользовательского домена у DNS-провайдера без добавления его на GitHub может привести к тому, что кто-то 
другой сможет разместить сайт на одном из ваших поддоменов.
Команда dig, которую можно использовать для проверки правильности настройки записей DNS, отсутствует в Windows. 
Чтобы проверить правильность настройки записей DNS, можно использовать Resolve-DnsNameкоманду PowerShell или установить BIND .
